## 一、流程概览

```
┌─────────────────────────────────────────────────────────────────┐
│  Phase 1: 问题发现   → Phase 2: 根因分析   → Phase 3: 方案设计  │
│  识别异常指标         追溯到代码层           填设计清单            │
└─────────────────────────────────────────────────────────────────┘
            ↓                   ↓                   ↓
┌─────────────────────────────────────────────────────────────────┐
│  Phase 4: 开发实现   → Phase 5: 验证测试     → Phase 6: 交付上线  │
│  小步提交Git          单元+集成+E2E           回归测试确认         │
└─────────────────────────────────────────────────────────────────┘
```

## 二、Phase 1: 问题发现

### 触发信号识别

| 信号类型 | 识别方法 | 示例 |
|----------|----------|------|
| **异常指标** | 关键指标异常偏离预期 | 89.5%连续买入、10笔交易 |
| **报错信息** | 代码运行报错 | ValueError、KeyError |
| **逻辑错误** | 行为不符合预期 | 止损不触发、重复买入 |

### 问题分类

| 类别 | 定义 | 处理方式 |
|------|------|----------|
| **P0** | 影响核心功能（如89.5%连续买入） | 立即停，优先修复 |
| **P1** | 影响部分功能 | 计划内修复 |
| **P2** | 非关键问题 | 记录，跟随版本修复 |

### 关键原则

**停止点判断**: 发现以下情况 → 停止 → 等确认
- 跨模块变更
- 两个相同功能的实现（重复实现）
- 无设计文档的新功能
- 不确定验收标准

## 三、Phase 2: 根因分析

### 根因分析方法

| 方法 | 说明 | 适用场景 |
|------|------|----------|
| **5-Why** | 连续追问5个为什么 | 快速定位 |
| **鱼骨图** | 人/机/料/法/环分析 | 复杂问题 |
| **假设验证** | 提出假设→验证→排除 | 不确定根因 |

### 根因模板

```markdown
## 根因分析报告

### 问题描述
[描述具体问题]

### 5-Why 追溯
- Why 1: [直接原因]
- Why 2: [上一层原因]
- Why 3: [更深层原因]
- Why 4: [系统层面原因]
- Why 5: [根因]

### 根因结论
[一句话总结根因]

### 类似问题扫描
[检查是否有其他模块存在同样问题]
```

### 案例：89.5%连续买入问题

```markdown
## 根因分析：89.5%连续买入

### 问题
回测中89.5%的买入发生在平仓当天（次日再买）

### 5-Why 追溯
1. Why: 为什么平仓后立即买入？
   → 因为没有"closed_today"机制

2. Why: 为什么没有closed_today？
   → 设计时没有考虑这个场景

3. Why: 为什么设计时遗漏？
   → 没有填"设计清单"（持仓管理机制）

4. Why: 为什么没有填设计清单？
   → 不知道有这个要求（隐性约束）

5. Why: 为什么是隐性约束？
   → 之前没遇到过类似问题，没有经验沉淀

### 根因
**设计时没有填"设计清单"，导致持仓管理机制缺失**

### 类似问题扫描
- [ ] 其他回测引擎是否有相同问题？
- [ ] 日内交易逻辑是否完整？
```

## 四、Phase 3: 方案设计

### 核心原则

> **先设计后实现，涉及多模块必须先输出设计文档**

### 设计清单（必须检查）

涉及回测/交易/持仓功能时，必须回答以下4项：

| # | 检查项 | 说明 |
|---|--------|------|
| **1** | 持仓管理机制 | 如何防止重复买入？closed_today？ |
| **2** | 交易执行模型 | T+1？当日？限价？市价？ |
| **3** | 止盈止损顺序 | 哪个先判断？优先级？ |
| **4** | min/max持仓天数 | 约束是什么？边界条件？ |

**缺失任一项 → 停止 → 补设计**

### 设计文档模板

```markdown
# [功能名称] 设计文档 v1.0

## 1. 问题背景
[为什么要改？当前问题是什么？]

## 2. 设计清单检查

| # | 检查项 | 答案 |
|---|--------|------|
| 1 | 持仓管理机制 | [具体方案] |
| 2 | 交易执行模型 | [T+1/当日？] |
| 3 | 止盈止损顺序 | [优先级] |
| 4 | min/max持仓天数 | [具体约束] |

## 3. 改动范围
[涉及哪些模块？]

## 4. 接口契约
[模块间怎么通信？]

## 5. 验收标准
[怎么判断改好了？]
- [ ] 测试用例1
- [ ] 测试用例2

## 6. 回滚方案
[如果失败怎么恢复？]

## 7. 风险控制
[有哪些风险？怎么缓解？]
```

## 五、Phase 4: 开发实现

### 小步提交原则

每个有意义的修改立即提交Git：
- ✅ 单个功能完成
- ✅ 测试用例通过
- ✅ 文档更新
- ❌ 不要憋一大坨代码

```bash
# 好的提交
git commit -m "feat: 添加closed_today机制防止重复买入"

# 不好的提交
git commit -m "修复了很多bug"
```

### 代码逻辑顺序规范

**优先级**: 止损（任意时刻）> 止盈（需min_days）> 到期

```python
# ✅ 正确
if hold_days >= min_days:
    if pnl >= stop_profit:
        sell()
elif hold_days >= max_days:
    sell()
else:
    if pnl <= stop_loss:
        sell()

# ❌ 错误：先判断止盈，再检查持仓天数
if pnl >= stop_profit:
    sell()  # 任意时刻触发，可能在min_days前
elif hold_days >= min_days:
    if pnl >= stop_profit:
        sell()  # 永远不会执行
```

### 硬编码检查

首次出现常量（API地址、超时、路径、文件名）就提取到 `constants.py`：

```python
# ❌ 错误
requests.get("https://qt.gtimg.cn/q=sh510300", timeout=10)

# ✅ 正确
from src.constants import TENCENT_BASE_URL, HTTP_TIMEOUT_SHORT
requests.get(f"{TENCENT_BASE_URL}/q=sh510300", timeout=HTTP_TIMEOUT_SHORT)
```

## 六、Phase 5: 验证测试

### 数据库变更检查清单

每次涉及 `daily` 表或其他核心表结构变更时：

```
[ ] 修改 schema/*.sql（版本控制）
[ ] 执行 scripts/init_database.py（迁移）
[ ] 代码启动时验证列是否存在
[ ] 单元测试覆盖
```

**教训**：schema与代码不一致会导致采集脚本全部失败（P0问题）

### 工具更新同步检查清单

每次新增/删除/合并脚本后，必须检查以下文档：

```
[ ] docs/TOOLS.md - 工具清单（工具名称、Usage、状态）
[ ] docs/INDEX.md - 场景索引（脚本列表）
[ ] docs/CHECK_REPORT.md - scripts目录一致性
[ ] docs/MODULES.md - 模块依赖（接口变更）
```

**教训**：工具更新不同步文档，导致使用者找不到正确工具

### 文档更新同步检查清单

每次新增/删除/修改文档后，必须更新 `docs/INDEX.md` 的"文档资产清单"部分：

```
[ ] 更新 INDEX.md 中的文档状态（活跃/废弃）
[ ] 更新"使用者"字段（哪些代码引用了这个文档）
[ ] 添加 metadata 头部（如需）
[ ] 更新"最后审查"日期
```

**文档 metadata 结构**：
```yaml
---
file: SOP-06.md
purpose: 标准化用户手动记录 ETF 买卖操作
used_by:
  - src/cli/decision.py
  - src/trade/tracker.py
status: active
last_review: 2026-06-07
review_interval: weekly
---
```

**教训**：文档没有"被谁使用"的声明，AI 无法知道哪些文档在用、哪些过期（US-014 复盘）

### 测试覆盖率要求

| 模块 | 覆盖率要求 |
|------|------------|
| 核心模块（持仓管理、风控） | ≥90% |
| 策略层 | ≥80% |
| 整体项目 | ≥80% |

### 关键路径测试清单

| # | 测试场景 | 边界条件 |
|---|----------|----------|
| **1** | 持仓管理：开仓→平仓→再开仓（同日） | closed_today检查 |
| **2** | 止盈触发 | min_days-1, min_days, min_days+1 |
| **3** | 止损触发 | 任意时刻 |
| **4** | 持仓天数到期 | =max_days, >max_days |
| **5** | 信号匹配 | 日期/代码匹配正确性 |

### 边界条件测试

```python
# 止盈边界测试
def test_stop_profit_at_min_days():
    """止盈在min_days刚好到达时触发"""
    position = create_position(hold_days=3)  # min_days=3
    pnl = 0.06  # 超过止盈阈值5.5%
    
    result = check_exit(position, pnl)
    assert result.reason == "stop_profit"

def test_stop_profit_before_min_days():
    """止盈在min_days之前不触发"""
    position = create_position(hold_days=2)  # min_days=3
    pnl = 0.06  # 超过止盈阈值5.5%
    
    result = check_exit(position, pnl)
    assert result is None  # 不触发
```

### 测试通过标准

| 测试类型 | 必须通过 |
|----------|----------|
| 单元测试 | ✅ 100% |
| 集成测试 | ✅ 100% |
| 回归测试 | ✅ 100% |
| E2E测试 | ✅ 100% |

**任一测试失败 → 停止 → 修复 → 重新验证**

## 八、常见问题处理

### 问题1：两个相同功能并存

**识别**：grep搜索发现两处相似实现

**处理**：
1. 保留更合理的实现
2. 删除冗余实现
3. 添加回归测试

### 问题2：测试覆盖不足

**识别**：skip测试长期存在

**处理**：
1. skip测试最长保留1周
2. 到期必须修复或删除
3. 补充边界条件测试

### 问题3：逻辑顺序错误

**识别**：止盈在持仓天数检查之前

**处理**：
1. 绘制执行流程图
2. 明确优先级约束
3. 单元测试覆盖所有分支

## 十、开发后自评（强制执行）

每次开发完成后，**必须填写自评**，作为交付的一部分。

### 自评清单

| # | 检查项 | 满分 | 实际得分 | 扣分原因 |
|---|--------|:----:|:--------:|----------|
| 1 | 设计文档输出（Phase 3） | 20 | | |
| 2 | 调研参考来源明确 | 20 | | |
| 3 | 按 SOP Phase 执行 | 20 | | |
| 4 | 单元测试覆盖核心路径 | 15 | | |
| 5 | 回归测试通过 | 15 | | |
| 6 | Git 小步提交 | 10 | | |
| **合计** | | **100** | | |

### 评分标准

| 分数 | 等级 | 说明 |
|:----:|------|------|
| 90-100 | 🏆 优秀 | 完全按 SOP 执行，无扣分项 |
| 70-89 | ✅ 合格 | 轻微偏差，可接受 |
| 50-69 | ⚠️ 需改进 | 明显偏差，需反思 |
| <50 | ❌ 不合格 | 违反 SOP，需要回滚重做 |

### 自评触发条件

**必须触发自评的场景**：
- 每次功能开发完成
- 每次 bug 修复完成
- 每次文档更新完成

**自评流程**：
```
开发完成 → 填写自评清单 → 计算得分 → 记录到 memory/YYYY-MM-DD.md → 如<70分立即反思
```

### 自评记录格式

```markdown
## 自评 - [功能名称] - [日期]

| # | 检查项 | 满分 | 得分 | 原因 |
|---|--------|:----:|:----:|------|
| 1 | 设计文档 | 20 | 20 | Phase 3 输出完整设计文档 |
| 2 | 调研来源 | 20 | 20 | 参考 leionion/ai-trading-journal-audit-tool |
| 3 | SOP Phase | 20 | 0 | ❌ 直接实现，未走 Phase 3 |
| 4 | 单元测试 | 15 | 10 | 仅手动测试，缺少自动化用例 |
| 5 | 回归测试 | 15 | 15 | history/eval 功能正常 |
| 6 | Git 提交 | 10 | 10 | 小步提交 |
| **合计** | | **100** | **75** | ⚠️ 需改进 |

### 低于70分的强制措施

1. **立即记录**：在 memory/YYYY-MM-DD.md 中标记"⚠️ 需改进"
2. **根因分析**：填写"扣分原因"列，找出行为偏差
3. **更新 SOP**：如果 SOP 本身有问题，更新 SOP
4. **防止重蹈**：将教训更新到 MEMORY.md
