## 4. 真实发现（最有价值的部分）

### 发现 1：D-013.2 漂移债是误诊

**现象**：80 天数据下，方案 A（510300 单点）只翻 2 次，平均段长 10 天。

**根因**：
- 用户原描述基于"印象/历史决策记录"，不是数据
- 实际 510300 在 2026-04-30 进入 trend_up 后**连续 33 天稳态**

**教训**（L339）：债项必须先验真伪再修，不能凭印象/历史描述直接改代码。

### 发现 2：core 池与"市场整体趋势"语义不匹配

**现象**：方案 C（14 ETF 众数 70%）在 2 年数据中 trend_up 召回仅 6.2~11.4%。

**根因分析**：
- core 池 14 只 ETF 行业分散（半导体/有色/传媒/银行/医药/科技...）
- 2024-2026 A 股明显"行业轮动"，同 mode 一致性低
- 510300 是大盘指数，能反映整体趋势，但 core 池的均值不能

**教训**（L340）：业界"多标投票"假设在 A 股 ETF 池场景不成立。**池设计决定方案可行性**。

### 发现 3：label 窗口/阈值决定方案排序

**现象**：60d/±3% 标签下 C 方案误判 71.9%，20d/±1.5% 下误判 31.2%。

**根因**：
- 60d/±3% 窗口长、阈值高 → 标签"稀疏"（41% trend_up / 56% range）→ C 方案少量 trend_up 命中即被放大误判
- 20d/±1.5% 窗口短、阈值低 → 标签"密集"（54% / 27% / 19%）→ C 方案的"保守"特性被奖励

**教训**（L341）：**改 label 之前不下方案结论**。任何对比必须先确定 label 合理性。

### 发现 4（次要）：浏览器/curl 国外网站必须显式 socks5 代理

**现象**：Google/DDG/Wikipedia 全 timeout。

**根因**：
- SOUL.md 规则提到"socks5://127.0.0.1:1080"
- 但**没自动生效**（`env | grep proxy` 空）
- 需 `browser_use` 传 `browser_args="--proxy-server=..."` 或 `curl --socks5`

**教训**：工具配置**必须验证**（规则 9），不能假设声明 = 生效。
